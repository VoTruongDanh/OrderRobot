# AI Memory

- Project: demo `Order Robot` with 3 services: `kiosk-ui`, `core-backend`, `ai-backend`.
- Current stack: React/Vite/TypeScript for UI, FastAPI for both backends, CSV for v1 data.
- Baseline completed:
  - UI kiosk with robot avatar, transcript, menu board, camera preview, and browser speech hooks.
  - Core backend reads `data/menu.csv`, writes `data/orders.csv`, and uses repository abstraction.
  - AI backend manages sessions, recommendations, cart confirmation, and OpenAI-compatible provider calls.
- Enhance round 1 completed:
  - Browser capability issues are now surfaced as non-fatal notices instead of blocking the kiosk flow.
  - UI now shows notices for missing `FaceDetector`, camera permission, microphone issues, and missing TTS/STT support.
  - AI backend now prunes expired sessions and exposes health metadata.
  - Conversation engine can add multiple menu items from one utterance, such as `1 tra dao va 1 matcha`.
  - Keyword matching was tightened to reduce false positives from substring collisions.
- Enhance round 2 completed:
  - Camera presence detection no longer depends on native browser `FaceDetector`.
  - `kiosk-ui` now uses `@tensorflow-models/blazeface` with local `tfjs` runtime to detect one face from the live camera stream.
  - Manual start remains available only as a fallback when camera permission or detector initialization fails.
- Enhance round 3 completed:
  - The kiosk layout is now tuned for portrait and tall-window displays, with the hero stack, conversation panel, camera strip, and menu grid rebalanced for vertical scanning.
  - Camera preview now moves ahead of supporting copy in portrait mode, and menu cards stay at two columns on tall kiosks before collapsing to one on narrow mobile widths.
  - Action buttons and status chips were tightened so controls remain readable without wasting vertical space.
- Enhance round 4 completed:
  - The kiosk UI was simplified to focus on only the essential demo sections: robot, conversation controls, transcript, cart, compact camera preview, and menu.
  - Status indicators were compressed into a single pill strip, notices were reduced to one primary message, and the menu cards now show only the fastest-to-scan information.
  - The separate camera area was reduced to a compact card so it stops competing visually with ordering and conversation.
- Enhance round 5 completed:
  - This round previously improved browser TTS selection, but the speech path has since moved to backend-first in round 6.
- Enhance round 6 completed:
  - Speech is now backend-first: the browser only records audio, sends it to `ai-backend`, and plays audio blobs returned by `ai-backend`.
  - This round first moved speech off browser APIs; the speech engines were later localized further in round 7.
- Enhance round 7 completed:
  - `AI_API_KEY` is now reserved for chat and decision-making, while TTS and STT are handled locally inside `ai-backend`.
  - `ai-backend` speech synthesis now uses local `pyttsx3`, and transcription now uses local `faster-whisper`.
  - The UI speech hook remained backend-first, so no browser speech engine is required beyond microphone capture and audio playback.

- Debug round 1 completed:

  - `ai-backend` now falls back to local Windows `System.Speech` synthesis through PowerShell whenever the `pyttsx3` driver stack fails with common Windows voice/runtime errors.
  - `pywin32` was also added to the backend requirements so the primary `pyttsx3` path can still work when the environment is installed cleanly.

- Enhance round 17 completed:
  - Fallback conversation text in `ai-backend` was rewritten to proper Vietnamese with diacritics, so transcript bubbles no longer show robot copy without accents.
  - The provider system prompt now explicitly requires Vietnamese output with natural diacritics, keeping provider-backed replies aligned with the local fallback style.
  - A backend test now locks the greeting text to accented Vietnamese to prevent regression.
- Enhance round 18 completed:
  - The menu board now shows local dish thumbnails for each current CSV item, using stable `item_id -> local image` mapping so the demo does not depend on external URLs.

  - The kiosk screen now hard-locks the main stage into a fixed top/bottom split on desktop so `robot + chat` stay in the upper half and the menu stays below instead of being pushed away by long transcripts.

- Enhance round 21 completed:
  - The frontend speech hook now ignores obvious ambient/watermark transcripts such as `subscribe`, `kênh`, `video hấp dẫn`, and repeated identical phrases within a short window.
  - Microphone capture was tightened with `autoGainControl` in addition to echo cancellation and noise suppression to reduce stray environmental audio being promoted into customer turns.

- Enhance round 22 completed:
  - Local STT in `ai-backend` now uses stronger decode defaults for Vietnamese ordering speech: `medium` model by default, wider beam search, `best_of`, and a longer VAD silence window.
  - The speech service now injects a prompt biased by the real menu items from `core-backend`, helping `faster-whisper` recognize in-domain drink names instead of drifting toward generic speech.
  - A post-processing layer now repairs near-miss drink phrases such as `cafe mui` into `Cà phê muối` and rejects ultra-short low-information noise such as `Bui` before it reaches the conversation engine.
- Enhance round 23 completed:
  - STT latency was reduced without changing the Whisper model by preloading the local `faster-whisper` model at `ai-backend` startup, avoiding the slow first live transcription hit.
  - Each transcription pass now streams uploaded audio from memory and disables timestamp generation, removing temporary-file overhead and extra decode work while preserving transcript quality.
  - The kiosk speech hook now auto-stops recording after a short silence window once the customer finishes speaking, so normal ordering turns no longer wait near the full 6.5 second cap.
- Enhance round 24 completed:
  - Order completion in `kiosk-ui` now opens a large success popup with full order detail, a locally rendered QR code, and a visible 6 second auto-close countdown instead of only showing a small inline invoice block.
  - The success popup blocks auto-start of the next camera session while it is visible, then resets the kiosk back to a clean waiting state for the next customer after closing.
  - QR generation is now local in the frontend using the order id, timestamp, totals, and line items, so the kiosk does not depend on an external QR API during demos.
- Enhance round 25 completed:
  - `ai-backend` no longer blocks application startup while preloading the Whisper STT model; preload now runs in a background task with error logging instead of holding FastAPI at `Waiting for application startup`.
  - This restores the normal kiosk greeting/session flow immediately after launch, while STT can still warm in the background for later voice turns.
- Enhance round 26 completed:
  - STT latency for continuous kiosk use was tightened further without changing the Whisper model by preloading not only the model but also the STT prompt, hotword list, and ordering lexicon caches before the first live transcript.
  - The speech service now keeps the decode prompt shorter and moves menu-driven domain bias into `hotwords`, reducing per-request decoding overhead while preserving drink-name accuracy.
  - The frontend recorder now emits fewer media chunks and stops based on post-speech silence after actual voice onset instead of a more conservative fixed recording window, which reduces turn-end waiting for short orders.
  - STT CPU threads now default to a bounded explicit value on this machine class instead of `0`, avoiding overly ambiguous runtime defaults and helping throughput stay steadier during repeated transcriptions.
- Enhance round 27 completed:
  - The speech path now supports websocket-based streaming STT messages between `kiosk-ui` and `ai-backend`, so the browser can push live audio chunks while speaking instead of waiting until the end to begin all backend work.
  - Streaming partial transcripts are now surfaced during listening for faster perceived response, while the final transcript still uses the full-quality transcription path before being sent into the ordering flow.
  - Empty websocket finalization now returns the same soft `retry` state as the regular STT endpoint, keeping the voice UX consistent.

- Debug round 2 completed:
  - Root cause for `dev:ai` randomly shutting down during `npm run dev` was the backend `uvicorn --reload` process watching the entire repo, so frontend lockfile and dependency churn could trigger backend reloads unrelated to `ai-backend`.
  - `ai-backend` was especially vulnerable because its startup now includes background preload work; when an unrelated repo-wide reload landed during that window, the process could get stuck waiting on background tasks and then exit with code 1.
  - The root `dev:core` and `dev:ai` scripts now scope `--reload-dir` to their own backend folders plus `data`, which keeps hot reload useful without letting frontend file changes destabilize backend dev servers.

- Remaining tech debt:
  - No browser e2e test yet for camera + speech behavior.
  - No provider fallback mode yet when API config is wrong but the user still wants to keep demoing.
  - MySQL adapter is still pending; only the interface boundary is ready.
  - Face detection now depends on `tfjs` performance on the kiosk machine, so lower-end devices may still need polling or resolution tuning.
  - Portrait optimization is CSS-driven; if a future kiosk shell changes DOM order or adds more modules, the portrait breakpoints should be reviewed again.
  - If users later need richer menu exploration, the simplified cards may need an optional detail view instead of restoring all metadata inline.
  - Local STT model loading can take time the first time `faster-whisper` downloads or warms the selected model.
  - The silence-based auto-stop is tuned conservatively, but very quiet microphones or unusually noisy kiosks may still need threshold tuning per device.
  - The success QR currently encodes order metadata as JSON for kiosk-side confirmation, not a public order lookup URL.
  - Local TTS quality depends on the voices installed on the Windows machine that runs `ai-backend`.
  - The invoice currently reuses the saved order payload; if a richer receipt format is needed later, a dedicated printable receipt schema or endpoint would still help.
